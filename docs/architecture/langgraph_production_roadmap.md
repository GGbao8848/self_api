# LangGraph 对接与生产化路线图

本文档记录 `self_api` 面向 LangGraph 的分期规划，作为后续架构演进基线。

## 目标

- 保持 `FastAPI` 作为稳定业务执行层
- 让 LangGraph 以 HTTP 工具方式稳定调用
- 逐步补齐跨机器部署、任务恢复、多副本扩展能力

## 分期

### 一期：最小生产版

目标：支持跨机器生产接入，但暂不解决任务崩溃自动恢复。

范围：

- 启用正式鉴权配置
- 明确共享存储与 `SELF_API_FILE_ACCESS_ROOTS`
- 为异步任务返回外部可访问的 `status_url`
- 增加生产就绪检查与部署文档
- 约定 LangGraph 长任务默认调用 `/async` 接口

非目标：

- 不引入任务队列
- 不引入数据库持久化任务状态
- 不做对象存储输入输出改造

### 二期：可恢复版

目标：API 重启、worker 崩溃后，任务状态不丢，任务可重试或可恢复判定。

范围：

- 任务状态迁移到数据库
- API 与执行 worker 解耦
- 引入 Redis/队列
- 增加任务心跳、租约、重试与回调补偿
- 增加幂等键

### 三期：多机多副本版

目标：FastAPI 多实例、worker 多实例、共享存储/对象存储。

范围：

- API 无状态化
- artifact 元数据入库
- 文件存储切换为共享存储或对象存储
- 增加资源治理与可观测性

## 一期落地约束

- 当前跨机器模式依赖共享文件系统，LangGraph 与 `self_api` 需要访问同一批目录
- 长任务统一使用 `/async` + `status_url` 或任务查询接口
- `SELF_API_PUBLIC_BASE_URL` 用于生成可被外部编排系统访问的任务状态地址

## 后续入库原则

- 规划变更先更新本文档，再推进实现
- 二期开始前需要先确定 PostgreSQL 与 Redis 选型
